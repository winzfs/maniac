# Maniac Garage 어드민 관리 기능 기획 및 개발 가이드

## 1. 문서 목적

이 문서는 `maniac` 프로젝트의 웹서비스 운영을 위한 어드민 기능을 체계적으로 설계하기 위해 작성한다.

Maniac Garage는 공개 장비 페이지, 사용자 생성 콘텐츠, 이미지 업로드, 정비 기록, 부품 정보, 향후 결제/판매용 페이지/커뮤니티 기능까지 포함할 가능성이 있다. 따라서 초기 MVP 단계부터 어드민 기능의 권한 구조, 운영 화면, 감사 로그, 콘텐츠 관리 원칙을 명확히 두어야 한다.

어드민은 단순한 관리자 페이지가 아니라 서비스 운영 안정성, 사용자 보호, 콘텐츠 품질, 결제/신고 대응, 장애 대응을 위한 내부 운영 도구로 설계한다.

---

## 2. 어드민 기능의 핵심 목표

어드민 기능은 아래 목적을 가진다.

1. 사용자와 장비 데이터를 안전하게 조회하고 관리한다.
2. 공개 페이지와 사용자 생성 콘텐츠의 품질을 관리한다.
3. 신고/숨김/제재/복구 같은 운영 절차를 체계화한다.
4. 프리미엄 스킨, 배지, 공지, 카테고리 같은 서비스 설정을 코드 수정 없이 관리한다.
5. 결제, 환불, 유료 기능 상태를 추적하고 대응할 수 있게 한다.
6. 모든 중요한 관리자 행동을 감사 로그로 남긴다.
7. Cloudflare 기반 배포 환경에서도 안전하게 접근 제어와 운영 작업을 수행한다.

## 3. 기본 원칙

### 3.1 어드민 기능은 MVP부터 구조만 잡는다

MVP에서 모든 어드민 기능을 완성할 필요는 없다. 하지만 아래 구조는 초기에 반드시 고려한다.

- 관리자 권한 모델
- 관리자 라우트 분리
- 관리자 접근 제어
- 감사 로그 테이블
- 사용자/장비/정비 기록 조회 화면
- 공개 콘텐츠 숨김 처리

초기에는 조회와 최소한의 숨김/복구 기능만 있어도 된다. 다만 나중에 기능을 붙일 수 있도록 도메인과 권한 구조를 먼저 설계한다.

### 3.2 일반 사용자 기능과 어드민 기능은 분리한다

어드민 기능은 일반 사용자 대시보드와 섞지 않는다.

권장 라우트:

```txt
/admin
/admin/users
/admin/equipments
/admin/maintenance-logs
/admin/photos
/admin/reports
/admin/settings
/admin/billing
/admin/audit-logs
```

주의:

- 일반 사용자 컴포넌트를 그대로 가져와 관리자 작업 버튼만 붙이는 방식을 피한다.
- 공개 페이지 조회 쿼리와 어드민 조회 쿼리를 분리한다.
- 어드민 API는 일반 API와 route/action/service 계층에서 명확히 구분한다.

### 3.3 모든 관리자 작업은 기록한다

어드민에서 데이터를 변경하는 행위는 반드시 감사 로그에 남긴다.

예:

- 사용자 상태 변경
- 장비 공개/비공개 변경
- 장비 페이지 숨김
- 사진 삭제/숨김
- 신고 처리
- 결제 상태 수동 변경
- 스킨/배지/공지 수정
- 관리자 권한 변경

감사 로그에는 최소한 아래 정보를 남긴다.

- 작업자 adminUserId
- 작업 종류 action
- 대상 리소스 resourceType
- 대상 ID resourceId
- 변경 전 값 beforeValue
- 변경 후 값 afterValue
- 사유 reason
- IP/userAgent
- createdAt

## 4. 관리자 권한 모델

어드민은 역할 기반 권한 제어, 즉 RBAC를 기본으로 한다.

### 4.1 권장 역할

#### Owner

서비스 소유자. 모든 권한을 가진다.

권한:

- 관리자 계정 관리
- 모든 데이터 조회/수정
- 결제/환불 관리
- 서비스 설정 변경
- 운영 정책 변경
- 위험 작업 승인

#### Admin

일반 관리자. 대부분의 운영 기능을 사용할 수 있다.

권한:

- 사용자 조회
- 장비/콘텐츠 관리
- 신고 처리
- 공지 관리
- 카테고리/스킨 관리
- 기본 통계 조회

제한:

- Owner 권한 부여 불가
- 결제 수동 변경은 제한 또는 승인 필요
- DB 위험 작업 불가

#### Moderator

콘텐츠 운영 담당자.

권한:

- 공개 장비 페이지 조회
- 신고 목록 조회
- 부적절한 콘텐츠 숨김
- 댓글/커뮤니티 기능 도입 시 게시물 숨김
- 사용자 경고 처리

제한:

- 결제/유저 개인정보/관리자 설정 접근 불가
- 영구 삭제 불가

#### Support

고객 응대 담당자.

권한:

- 사용자 기본 정보 조회
- 결제 상태 조회
- 문의 대응에 필요한 장비/기록 조회
- 계정 상태 확인

제한:

- 데이터 수정 최소화
- 콘텐츠 숨김/삭제 불가 또는 제한
- 개인정보 전체 노출 제한

#### Viewer

읽기 전용 관리자.

권한:

- 기본 통계 조회
- 공개 콘텐츠 조회

제한:

- 수정/삭제/제재 불가

### 4.2 권한 설계 원칙

- 역할은 문자열 하드코딩보다 enum 또는 DB 기반으로 관리한다.
- `isAdmin` 하나로 모든 권한을 처리하지 않는다.
- 권한 체크는 UI에서만 하지 말고 서버 action/API에서 반드시 재검증한다.
- 위험 작업은 Owner 또는 2단계 확인을 요구한다.
- 관리자 권한 변경은 반드시 감사 로그에 남긴다.

## 5. 어드민 도메인 모델

### 5.1 AdminUser 또는 UserRole

기존 `users` 테이블에 role을 직접 넣을 수도 있지만, 장기적으로는 별도 권한 테이블을 권장한다.

예시:

```txt
user_roles
- id
- user_id
- role
- granted_by
- granted_at
- revoked_at
- created_at
```

주의:

- 관리자 권한은 soft revoke를 고려한다.
- 누가 언제 권한을 부여/회수했는지 남긴다.
- 운영 DB에서 수동으로 role을 수정하지 않는다.

### 5.2 AdminAuditLog

관리자 작업 로그.

```txt
admin_audit_logs
- id
- admin_user_id
- action
- resource_type
- resource_id
- before_value_json
- after_value_json
- reason
- ip_address
- user_agent
- created_at
```

주의:

- 감사 로그는 일반 관리자가 삭제할 수 없게 한다.
- 로그 조회는 Owner/Admin 일부 권한으로 제한한다.
- 개인정보가 과도하게 저장되지 않도록 before/after 값 범위를 조절한다.

### 5.3 Report

사용자 신고.

```txt
reports
- id
- reporter_user_id
- target_type
- target_id
- reason_code
- reason_text
- status
- assigned_admin_id
- resolved_by
- resolved_at
- resolution_note
- created_at
```

상태 예시:

```txt
pending
reviewing
resolved_hidden
resolved_rejected
resolved_warning
resolved_banned
```

### 5.4 ModerationAction

신고나 관리자 판단으로 콘텐츠에 취한 조치.

```txt
moderation_actions
- id
- admin_user_id
- target_type
- target_id
- action
- reason
- created_at
```

조치 예시:

```txt
hide
unhide
remove
warn_user
suspend_user
restore
```

### 5.5 SiteSetting

서비스 운영 설정.

```txt
site_settings
- id
- key
- value_json
- description
- updated_by
- updated_at
```

사용 예:

- 가입 허용 여부
- 점검 모드
- 업로드 최대 용량
- 기본 공개 여부
- 신고 자동 숨김 기준
- 판매용 페이지 가격

주의:

- 모든 설정을 DB로 빼면 복잡해진다.
- 자주 바뀌는 운영 설정만 DB로 관리한다.
- 코드 안정성에 영향을 주는 핵심 설정은 feature flag 또는 환경변수로 관리한다.

## 6. 어드민 화면 구성

### 6.1 어드민 대시보드

운영자가 서비스 상태를 빠르게 볼 수 있는 첫 화면.

표시 항목:

- 총 사용자 수
- 신규 가입자 수
- 총 장비 수
- 공개 장비 수
- 정비 기록 수
- 사진 업로드 수
- 신고 대기 건수
- 최근 관리자 작업
- 최근 에러/운영 알림
- R2 저장소 사용량
- Cloudflare 트래픽/캐시 상태 요약

MVP에서는 간단한 숫자 카드부터 시작한다.

### 6.2 사용자 관리

기능:

- 사용자 목록
- 이메일/닉네임 검색
- 가입일/최근 활동일 필터
- 상태 필터: active, suspended, deleted
- 사용자 상세 조회
- 사용자의 장비 목록 조회
- 사용자의 신고/제재 이력 조회
- 계정 정지/복구
- 관리자 메모 작성

주의:

- 개인정보는 필요한 만큼만 노출한다.
- 이메일, OAuth ID 등 민감 정보는 권한에 따라 마스킹한다.
- 계정 삭제/정지는 즉시 수행보다 확인 모달과 사유 입력을 요구한다.

### 6.3 장비 관리

기능:

- 장비 목록
- 카테고리/공개 상태/생성일 필터
- 장비명/브랜드/모델/사용자 검색
- 장비 상세 조회
- 공개 페이지 미리보기
- 장비 숨김/복구
- 대표 이미지 확인
- 연결된 정비 기록/부품/사진 조회

주의:

- 관리자도 사용자의 비공개 장비를 함부로 공개 전환하지 않는다.
- 숨김은 서비스 운영 차원의 조치이고, 사용자 공개 설정과 구분한다.
- 운영 숨김 상태는 `moderationStatus` 같은 별도 필드로 관리한다.

예:

```txt
visibility: public | private
moderationStatus: normal | hidden | removed
```

### 6.4 정비 기록 관리

기능:

- 정비 기록 목록
- 장비/사용자/기록 유형별 필터
- 공개 여부 필터
- 부적절한 기록 숨김
- 신고된 기록 검토

주의:

- 정비 기록은 사용자의 개인 기록 성격이 있으므로 공개된 기록 위주로 운영 관리한다.
- 비공개 기록 조회는 Support/Admin 이상 권한과 명확한 사유가 필요하다.
- 판매용 인증 페이지에 쓰인 기록은 수정/삭제 이력을 특히 주의한다.

### 6.5 사진/이미지 관리

기능:

- 최근 업로드 이미지 조회
- 신고된 이미지 검토
- 이미지 숨김/복구
- R2 object 경로 확인
- 썸네일 생성 상태 확인
- 손상 이미지/미사용 이미지 탐지

주의:

- 원본 삭제보다 숨김 처리 우선.
- R2 object 삭제는 DB 기록과 함께 처리한다.
- 이미지 URL이 외부로 공유될 수 있으므로 공개 접근 정책을 신중히 설계한다.

### 6.6 신고 관리

기능:

- 신고 목록
- 상태별 필터
- 신고 대상 미리보기
- 신고 사유 확인
- 처리 담당자 지정
- 숨김/경고/기각/정지 처리
- 처리 메모 작성

운영 흐름:

1. 신고 접수
2. 관리자 검토 시작
3. 대상 콘텐츠 확인
4. 조치 결정
5. 신고 상태 변경
6. 감사 로그 기록
7. 필요한 경우 사용자 알림

주의:

- 신고 처리 결과는 재검토 가능해야 한다.
- 자동 삭제보다 숨김 처리 우선.
- 같은 대상에 신고가 누적되면 우선순위를 높인다.

### 6.7 카테고리/정비 유형 관리

기능:

- 장비 카테고리 관리
- 카테고리별 기본 정비 유형 관리
- 카테고리별 사용량 단위 관리
- 부품 카테고리 관리
- 공개 여부/정렬 순서 관리

주의:

- 이미 사용 중인 카테고리 코드는 삭제하지 않는다.
- 비활성화 처리로 숨긴다.
- 코드값 변경은 기존 데이터와 호환성 문제가 생긴다.

### 6.8 스킨/배지/프리미엄 상품 관리

기능:

- 스킨 목록
- 스킨 미리보기 이미지
- 적용 가능 카테고리
- 무료/유료 여부
- 가격
- 판매 상태
- 배지 관리
- 사용자 구매 내역 조회

주의:

- 스킨 설정은 코드와 DB의 역할을 분리한다.
- 실제 렌더링 컴포넌트는 코드에 있고, 노출/가격/설명은 DB에서 관리한다.
- 이미 판매된 스킨은 삭제하지 말고 판매 중지 상태로 둔다.

### 6.9 공지/랜딩 콘텐츠 관리

기능:

- 공지사항 작성/수정/예약 게시
- 랜딩 페이지 일부 문구 관리
- FAQ 관리
- 약관/개인정보처리방침 링크 관리

주의:

- 랜딩 전체를 DB CMS로 만들 필요는 없다.
- 자주 바뀌는 운영성 문구만 관리한다.
- 약관/정책 문서는 버전 관리가 필요하다.

### 6.10 결제/구독 관리

Phase 3 이후 추가한다.

기능:

- 결제 내역 조회
- 구매 상품 조회
- 결제 상태 확인
- 환불 요청 상태 관리
- 웹훅 수신 로그 조회
- 유료 스킨/판매용 페이지 권한 확인
- 수동 권한 부여/회수

주의:

- 결제 상태 수동 변경은 Owner 또는 승인 절차가 필요하다.
- 결제 웹훅 원본 이벤트를 저장한다.
- 환불/취소/권한 회수는 감사 로그에 반드시 남긴다.

### 6.11 운영 로그/감사 로그

기능:

- 관리자 작업 로그 조회
- 관리자별 필터
- 리소스별 필터
- 작업 종류별 필터
- 기간 필터
- 상세 before/after 확인

주의:

- 감사 로그 삭제 기능은 제공하지 않는다.
- 개인정보가 과도하게 보이지 않도록 권한별 마스킹을 적용한다.

### 7.2 어드민 접근 체크

모든 어드민 페이지와 action은 서버에서 권한을 검증한다.

예시 흐름:

```txt
request
  -> getSessionUser()
  -> requireAdminRole(userId, requiredPermission)
  -> execute action/query
  -> write audit log if mutation
```

주의:

- UI에서 메뉴를 숨기는 것만으로는 보안이 아니다.
- 서버 action/API에서 매번 권한을 확인한다.
- 관리자 권한 캐시는 짧게 유지하거나 권한 변경 시 무효화한다.

### 7.3 권한 체크 예시

```ts
type AdminPermission =
  | 'user:read'
  | 'user:suspend'
  | 'equipment:read'
  | 'equipment:moderate'
  | 'report:read'
  | 'report:resolve'
  | 'setting:update'
  | 'billing:read'
  | 'billing:update'
  | 'audit:read';
```

권한 매핑은 설정 파일 또는 DB에서 관리한다.

```ts
const ROLE_PERMISSIONS = {
  owner: ['*'],
  admin: ['user:read', 'equipment:read', 'equipment:moderate', 'report:read', 'report:resolve', 'setting:update'],
  moderator: ['equipment:read', 'equipment:moderate', 'report:read', 'report:resolve'],
  support: ['user:read', 'equipment:read', 'billing:read'],
  viewer: ['equipment:read'],
};
```

## 8. Cloudflare 기반 어드민 운영 주의사항

### 8.1 어드민 라우트 보호

Cloudflare Pages/Workers 환경에서도 어드민 라우트는 반드시 애플리케이션 레벨 권한 체크를 통과해야 한다.

추가 보호 옵션:

- Cloudflare Access로 `/admin/*` 접근 제한
- 관리자 계정 2FA 강제
- 특정 국가/위험 IP 차단 규칙
- rate limiting

Cloudflare Access를 사용하더라도 앱 내부 RBAC는 반드시 유지한다.

### 8.2 Preview/Production 분리

- Preview 환경 어드민은 production DB를 절대 조작하지 않는다.
- Preview R2 bucket과 production R2 bucket을 분리한다.
- Preview에서 결제 웹훅/알림톡/운영 알림이 실제 발송되지 않게 한다.

### 8.3 운영 작업 제한

Edge 환경에서는 장시간 작업이 제한될 수 있다.

주의:

- 대량 이미지 정리
- 대량 데이터 내보내기
- 통계 재계산
- 대량 알림 발송

이런 작업은 Cron Trigger, Queue, 별도 batch worker로 분리한다.

### 8.4 캐시 주의

어드민 페이지는 캐시하지 않는다.

- `Cache-Control: no-store`
- 민감 데이터가 CDN에 캐시되지 않도록 한다.
- 공개 페이지 캐시와 어드민 API 캐시 정책을 분리한다.

## 9. 어드민 보안 원칙

- 관리자 로그인은 일반 사용자 로그인보다 강한 보안 정책을 적용한다.
- 가능한 경우 2FA를 요구한다.
- 관리자 세션 만료 시간을 짧게 둔다.
- 위험 작업은 사유 입력을 필수화한다.
- 영구 삭제는 최소화하고 soft delete/hide를 우선한다.
- 관리자 권한 변경은 Owner만 가능하게 한다.
- 모든 mutation은 감사 로그를 남긴다.
- 개인정보 노출은 최소화하고 권한별 마스킹을 적용한다.
- 다운로드/내보내기 기능은 별도 권한으로 분리한다.
- 운영자 본인 계정의 권한 상승을 직접 수행하지 못하게 한다.

## 14. 최종 방향

Maniac Garage의 어드민 기능은 초기부터 **운영 안정성, 권한 분리, 감사 로그, 콘텐츠 품질 관리**를 중심으로 설계한다.

초기 MVP에서는 모든 관리 기능을 구현하지 않아도 되지만, 아래 4가지는 반드시 구조에 포함한다.

1. 역할 기반 관리자 권한
2. 관리자 전용 route/action/query 분리
3. 모든 중요 mutation의 감사 로그
4. 콘텐츠 숨김/복구 중심의 moderation 구조

어드민은 나중에 붙이는 부가 기능이 아니라, 서비스가 공개 페이지와 사용자 콘텐츠를 다루기 위한 필수 운영 기반으로 본다.
